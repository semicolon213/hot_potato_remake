# 초기 로딩 정확한 파악 (코드 기준)

앱스크립트는 수정하지 않는 전제로, **언제·무엇이·어디에** 저장되는지만 기술합니다.

---

## 1. 트리거 조건

**파일:** `src/hooks/core/useAppState.ts` (256~257행)

```ts
if (user && user.isApproved && !isLoading && !isInitializingData && !hasInitialized)
```

- **실행 시점:** 로그인·승인 완료(`user.isApproved`), 인증 로딩 종료(`!isLoading`), 아직 초기화 미실행(`!hasInitialized`), 초기화 중이 아님(`!isInitializingData`)일 때 **1회**.
- **실행 내용:** `initAndFetch()` 호출 → 내부에서 `dataSyncService.initializeDataFull(user, onProgress)` 실행.

## 2. initializeDataFull 실행 순서

**파일:** `src/services/dataSyncService.ts` (258~326행)

| 단계 | 내용 | 비고 |
|------|------|------|
| 1 | `initializeSpreadsheetIds()` | 아래 3단계 참고 |

| 5 | 완료 후 `lastSyncTime` 갱신, `spreadsheetIds` 반환 | 반환값은 ID만, 실제 데이터는 캐시/전역 변수에 저장됨 |

## 4. INITIAL_DATA_TASKS 목록 및 runIf

**파일:** `src/services/dataSyncService.ts` (25~188행)

| key | name | scope | runIf | 저장 위치 (데이터 흐름) |
|-----|------|--------|--------|-------------------------|
| announcements | 공지사항 | global | `user.studentId && user.userType` | apiClient 캐시 (`getAnnouncements` 등). **UI 상태는 별도 useEffect에서 setAnnouncements** |
| calendar-events | 캘린더 이벤트 | global | true | apiClient 캐시 (`fetchCalendarEvents`). **UI 상태는 별도 useEffect에서 setCalendarEvents** |
| documents | 전체 문서 목록 | global | true | loadAllDocuments 내부에서 apiClient/캐시 사용 여부는 해당 함수 구현에 따름 |
| templates | 템플릿 목록 | global | true | apiClient 캐시 (`getTemplates`) |
| shared-templates | 공유 템플릿 | global | true | apiClient 캐시 |
| static-tags | 기본 태그 | global | true | apiClient 캐시 |
| workflow-templates | 워크플로우 템플릿 | global | true | apiClient 캐시 |
| students | 학생 목록 | page | true | papyrusManager 호출 → 캐시 여부는 해당 API 구현에 따름 |
| staff | 교직원 목록 | page | true | 동일 |
| attendees | 참석자 목록 | page | true | 동일 |
| ledger-list | 장부 목록 | global | true | apiClient 캐시 (`getLedgerList`) |
| my-requested-workflows | 내가 올린 결재 | dashboard | `user.email` | apiClient 캐시 |
| my-pending-workflows | 내 담당 워크플로우 | dashboard | `user.email` | apiClient 캐시 |
| completed-workflows | 완료된 워크플로우 | dashboard | `user.email` | apiClient 캐시 |
| all-users | 전체 사용자 | page | `user.isAdmin` | apiClient 캐시 |
| pending-users | 승인 대기 사용자 | dashboard | `user.isAdmin` | apiClient 캐시 |

- **중요:** 태스크 `fn`은 **반환값을 사용하지 않음** (`Promise.allSettled`만 사용).  
  실제 효과는 **apiClient.request / papyrusManager 등이 내부에서 캐시에 넣는 것**과, **papyrusManager 전역 변수**(스프레드시트 ID) 갱신.

## 5. 초기 로딩 직후 UI 상태(공지·캘린더·템플릿 등) 채우는 경로

**파일:** `src/hooks/core/useAppState.ts`

- `initializeDataFull`가 **끝난 뒤** 반환된 `spreadsheetIds`로  
  `setAnnouncementSpreadsheetId`, `setCalendarProfessorSpreadsheetId`, … `setHotPotatoDBSpreadsheetId` 등이 호출되고,  
  `setIsGapiReady(true)`, `setHasInitialized(true)` 로 앱이 “초기화 완료” 상태가 됨.

- **공지·캘린더 UI용 상태**는 `initializeDataFull`가 아니라 **별도 useEffect**에서 채워짐:
  - **공지:** 361~378행 — `isGapiReady && announcementSpreadsheetId && user?.studentId && user?.userType` 일 때  
    `fetchAnnouncements(...)` 호출 → **이때는 이미 초기 로딩 태스크에서 같은 API를 호출해 두어 캐시 히트** → `setAnnouncements(announcementsData)`.
  - **캘린더:** 380~398행 — `isGapiReady` 및 캘린더용 스프레드시트 ID 중 하나라도 있을 때  
    `fetchCalendarEvents()` 호출 → **캐시 히트** → `setCalendarEvents(events)`.
  - **템플릿/태그:** 400~422행 — `isGapiReady` 일 때 `fetchTemplates()`, `fetchPersonalTags()` 호출.  
    (참고: `papyrusManager.fetchTemplates`는 현재 빈 배열 반환; 템플릿은 `apiClient.getTemplates` 등 다른 경로로 캐시됨.)
  - **참석자(학생/교직원):** 424~439행 — `isGapiReady && studentSpreadsheetId && staffSpreadsheetId` 일 때 `fetchAttendees()` → `setStudents`, `setStaff`.

즉, **초기 로딩 = 캐시 채우기 + 스프레드시트 ID 설정**이고, **화면에 보이는 공지/캘린더/템플릿 등은 그 다음에 도는 useEffect가 같은 API를 한 번 더 호출(캐시 히트)해서 `useAppDataStore`/useAppState에 넣는 구조**입니다.

## 6. 데이터 저장 위치 요약

| 구분 | 저장소 | 채우는 쪽 | 사용하는 쪽 |
|------|--------|------------|-------------|
| 스프레드시트 ID들 | papyrusManager 전역 변수 + useAppState state | initializeSpreadsheetIds | 전역 변수 직접 참조 또는 useAppState에서 꺼내 쓰는 컴포넌트 |
| getLedgerList, getTemplates, getAnnouncements, fetchCalendarEvents 등 | apiClient → cacheManager (메모리 + localStorage) | initializeDataFull 내 태스크 fn 호출 시 API가 캐시에 저장 | 각 페이지/훅에서 동일 API 재호출 시 캐시 히트 |
| announcements, calendarEvents | useAppDataStore (Zustand) | useAppState의 별도 useEffect (fetchAnnouncements, fetchCalendarEvents 호출 후 setAnnouncements, setCalendarEvents) | 공지/캘린더 UI, 위젯 등 |
| customTemplates, tags, students, staff | useAppState 내부 state | 위와 같은 useAppState의 useEffect (fetchTemplates, fetchPersonalTags, fetchAttendees 등) | 해당 데이터를 쓰는 페이지/훅 |

## 7. 정리

- **트리거:** 로그인·승인 후 `hasInitialized === false` 일 때 1회.
- **실제 작업:**  
  1) `initializeSpreadsheetIds()` → getSpreadsheetIds 1회 + findPersonalConfigFile / getAccountingFolderId 병렬 1회.  
  2) `initializeDataFull` → 8개씩 배치로 태스크 실행, 각 태스크는 해당 API 호출만 하며 **캐시만 채움**.  
  3) 반환된 스프레드시트 ID로 useAppState의 ID state + `isGapiReady`/`hasInitialized` 설정.
- **UI 데이터:** 공지/캘린더/템플릿/참석자 등은 **초기 로딩이 끝난 뒤** useAppState의 **별도 useEffect**에서 같은 API를 다시 호출해 **캐시에서 읽고** setAnnouncements, setCalendarEvents, setCustomTemplates 등으로 채움.
- **장부 목록:** 초기 로딩 태스크 `ledger-list`가 `apiClient.getLedgerList()`를 호출해 캐시에 넣고, 장부 페이지의 `useLedgerManagement`는 캐시 우선 조회 후 없을 때만 API 호출하도록 이미 개선된 상태.

이 문서는 앱스크립트를 건드리지 않는 범위에서, **초기 로딩이 “언제, 무엇을, 어디에” 넣는지**만 정확히 파악한 내용입니다.
